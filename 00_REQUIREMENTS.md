# 00. 全体要件定義書 — ダーツ大会 ライブ進行＆VS演出システム

> 本書は「何を作るか・どこまで作るか・どの順番で作るか」を定義する。
> 技術詳細は `01_TECH_REQUIREMENTS.md`、見た目は `02_DESIGN_REQUIREMENTS.md`、
> Cursor運用ルールは `03_CURSOR_PROJECT_RULES.md` を参照。

---

## 1. プロジェクト概要

ダーツ大会（ダブルイリミネーション）の進行を、その日集まった参加者の顔写真・ペア・トーナメント配置を使って、テレビの対戦演出（M-1グランプリ／格闘技中継のVS画面）のように盛り上げながら自動進行するシステム。

放送と違うのは「事前収録ではなく、当日その場で確定したデータをリアルタイム合成して演出に流し込む」点。これを実現するための中核設計が **「演出テンプレートは事前固定・中身（顔写真／名前／座標）だけ実行時に差し込む」** という考え方である。試合ごとに動画を生成しない。これによりライブ進行が止まらない。

---

## 2. ゴールと成功条件

- 当日、運営がほぼボタン操作だけで大会を完走できる。
- 試合確定の瞬間に「線がぶつかる → 爆発 → 顔写真付きVS画面」が遅延なく流れる。
- どの試合でも演出のクオリティが一定（テンプレ駆動なので破綻しない）。
- 誤操作してもundoで復旧でき、進行が壊れない。
- ダブルイリミネーションの勝敗・敗者ブラケット落としが100%正しい。

---

## 3. 利用者と利用シーン

| ロール | 端末 | やること |
|---|---|---|
| 運営オペレーター | タブレット／PC（操作端末） | 参加者登録、抽選実行、勝敗入力、進行操作 |
| 観客・参加者 | 大型スクリーン／プロジェクター（表示端末） | トーナメント表・VS演出・爆発を見る |
| 司会（任意） | 表示端末を見ながら | 演出に合わせてアナウンス |

- 会場は1つ。インターネット接続あり（Supabase利用のため）。
- 表示端末は **Chromeキオスク1台に固定**（クロスブラウザ対応はやらない＝大幅な簡略化）。
- 操作端末と表示端末は別物。同期はSupabase Realtimeで行う。

---

## 4. 用語定義

| 用語 | 意味 |
|---|---|
| イベント / 大会 | その日の1開催単位 |
| 参加者 (Participant) | 顔写真＋名前を登録した個人 |
| チーム / ペア (Team) | ランダム抽選で組まれた競技単位（原則2名）。トーナメントの「枠」はこのチーム |
| 試合 (Match) | チーム vs チーム |
| ブラケット (Bracket) | トーナメント表全体 |
| WB (Winners Bracket) | 勝者側の山 |
| LB (Losers Bracket) | 敗者復活側の山 |
| GF (Grand Final) | WB勝者 vs LB勝者の決勝 |
| ノード (Node) | トーナメント表上の各チームの枠。画面上の x/y 座標を持つ |
| シード (Seed) | ブラケットへの初期配置順 |
| bye / 不戦勝 | 人数が2のべき乗でないときの空き枠による自動勝ち上がり |

## 5. システム構成（概念）

```
[操作端末: 運営UI] ──┐
                     ├──> [Supabase: 真実源(DB/Storage) + Realtime] ──> [表示端末: 演出フロント]
[カメラ撮影] ────────┘
```

- **真実源(Single Source of Truth)** はSupabase上のブラケット状態とイベントデータ。
- 操作端末が状態を更新 → Realtimeで表示端末へ通知 → 表示端末が演出を再生。
- 表示端末をリロードしても、真実源から状態を復元して同じ画面に戻れること（必須）。

---

## 6. 機能要件一覧

### 6.1 参加者登録
- カメラでその場の顔写真を撮影して登録。
- 撮影時にガイド枠を表示し、顔を自動トリミングして全員のアスペクト比を統一。
- 名前を入力。
- 登録済み一覧の表示・削除・撮り直し。

### 6.2 チーム編成（抽選）
- 参加者をランダムにシャッフルしてペア（原則2名）を生成。
- 奇数人数の処理ルール（§9参照）を適用。
- 抽選結果のやり直し（確定前のみ）。
- 抽選を「演出として見せる」モード（任意・Phase2では静的でも可）。

### 6.3 トーナメント生成
- チームを使ってダブルイリミネーションのブラケットを自動生成。
- byeの自動挿入。
- 各ノードの画面座標を自動計算（**座標駆動**。詳細は技術要件）。
- 名前と顔写真をノードに自動配置。

### 6.4 試合進行
- 「次の試合」を自動提示。
- 勝敗をボタンで入力（チームA勝ち / チームB勝ち）。
- 入力に応じてブラケット状態を更新（勝者は次へ、敗者はLBの正しい枠へ）。
- 直前の操作をundo。
- GFの2本目（ブラケットリセット）に対応（§9参照）。
- 大会終了（優勝確定）の検知と表示。

### 6.5 アナウンス＆演出（フロント）
- 試合確定時に、対象2ノードへ向けてトーナメント表の線が伸びて衝突。
- 衝突点で爆発（透過動画 or パーティクル）。
- 全画面VS演出：`⭕️⭕️ vs ◯◯` を顔写真・名前付きで表示。
- 演出はスキップ可能（リハ・時間短縮用）。
- 演出は座標駆動：当日の配置が変わっても自動追従する。

### 6.6 運営補助
- リハーサル / デモモード（ダミーデータで全演出を試せる）。
- 進行状況の俯瞰（残り試合数、現在の試合）。

---

## 7. 非機能要件

| 区分 | 要件 |
|---|---|
| ライブ性 | 試合確定〜VS表示までの体感遅延を最小化。重い生成処理を本番中に走らせない |
| プリロード | 背景動画・爆発動画・Lottie・顔写真は演出前に必ずプリロード済みであること |
| 復元性 | 表示端末・操作端末ともリロードで状態復元できること |
| 同期 | 操作端末の操作が表示端末へ概ね1秒以内に反映 |
| パフォーマンス | 表示端末は1080pで安定描画（GPU有効）。素材は最適化前提 |
| 動作環境 | 表示端末はChrome固定。操作端末はChrome/Safari想定 |
| 肖像データ | 当日限りの利用を前提とし、大会終了後に削除できる導線を持つ |
| 可用性 | 単一会場・単一イベントの運用。高可用性やマルチテナントは今回スコープ外 |

## 8. フェーズ分割（実装順）

各フェーズは「Doneの定義」を満たしたら次へ進む。**演出は最後に乗せる**（土台が壊れていると演出のデバッグが地獄になるため）。

### Phase 0 — 基盤
- リポジトリ作成（ideal合同会社 GitHub Org）、Vite+React+TS雛形、Supabaseプロジェクト、env設定、ルーティング、Realtime疎通テスト。
- **Done:** 操作端末で押したボタンが表示端末にRealtimeで届くダミー疎通が動く。

### Phase 1 — 参加者登録
- カメラ撮影、顔トリミング、Storage保存、名前入力、一覧/削除。
- **Done:** 顔写真付き参加者をDBに登録・一覧表示できる。

### Phase 2 — チーム編成＆トーナメント生成（座標まで）
- ランダムペア生成、奇数処理、ダブルイリミ生成、bye挿入、ノード座標の自動計算。
- **Done:** 演出ナシで、正しい構造のトーナメント表（線・ノード・顔写真）が静的に描画でき、各ノードの座標が取得できる。

### Phase 3 — 試合進行ロジック
- 次試合提示、勝敗入力、勝者前進、敗者LB落とし、GFリセット、undo、優勝検知。操作端末↔表示端末の同期。
- **Done:** 演出ナシで、ボタン操作だけで大会を最初から最後まで完走でき、ブラケットが常に正しい。

### Phase 4 — 静的トーナメント表示の作り込み
- WB/LBの配色分け、現在進行ノードのハイライト、レイアウトの見やすさ。
- **Done:** 表示端末で進行状況が一目で分かる静的トーナメント表が完成。

### Phase 5 — 演出レイヤー
- 線の伸長→衝突→爆発→全画面VSのGSAPタイムライン、透過動画/パーティクル、VS画面、スキップ。
- **Done:** 試合確定で一連の演出が座標駆動で再生される。

### Phase 6 — 仕上げ
- 全素材プリロード、キオスク設定、リハーサルモード、エラー復帰、本番リハ。
- **Done:** 当日と同条件のリハで通しが成功する。

## 9. 業務ルール・エッジケース

- **奇数人数:** いずれかのチームを3名にする／不足分をプレースホルダ（"幽霊"）にする／1名を見学にする、のいずれかを運営が選択できる。MVPでは「1チームを3名」または「bye扱い」を既定とする。
- **チーム数が2のべき乗でない:** byeを自動挿入してシードを埋める。
- **棄権・無効試合:** 勝敗入力の選択肢に「不戦勝/棄権」を持てると望ましい（Phase3以降の拡張）。
- **誤操作:** 直前1手のundoを必須機能とする（多段undoは任意）。
- **ダブルイリミの優勝条件:** WB勝者はGFに無敗で到達、LB勝者は1敗済み。GF第1試合でLB勝者が勝った場合は両者1敗となるため **ブラケットリセット（GF第2試合）** を行い、その勝者を優勝とする。WB勝者がGF第1試合に勝てばその時点で優勝。
- **同点/再試合:** 本システムは勝敗二択を前提。引き分けは扱わない。

---

## 10. スコープ外（今回はやらない）

- クロスブラウザ完全対応（表示はChrome固定）。
- マルチ会場・複数イベント同時進行・SaaS化（将来拡張余地としてデータ設計だけ意識）。
- 観客のスマホからの操作・投票。
- 高度な3Dフル演出（放送スタジオ品質の完全再現）。
- 課金・認証基盤（運営者限定の簡易運用）。
