# 03. Cursor プロジェクトルール

> 配置: `.cursor/rules/project.mdc`（推奨）または `.cursorrules` としてリポジトリ直下に置く。
> 目的: ライブ進行を壊す実装・演出破綻・状態崩壊を**未然に防ぐ**ためのガードレール。
> AIアシスタント(Cursor)は本ルールを最優先で遵守すること。

---

## 1. プロジェクトの絶対原則（これを破る提案はしない）

1. **ライブ進行を絶対に止めない。** 本番中に重い生成処理（動画レンダリング等）を走らせない。演出は「事前固定テンプレ＋実行時差し込み」で実現する。試合ごとに動画を生成しない。
2. **演出は座標駆動。** 衝突点・爆発位置・顔の表示位置は、レイアウト計算が返すノード座標から算出する。**座標やレイアウトをハードコードしない。**
3. **全メディアはプリロード後に再生。** 未ロードのまま動画/Lottie/顔写真を再生しない。プリロード完了をPromiseで待ってから演出を発火する。
4. **状態の真実源はSupabaseの `bracket_snapshot`。** UIローカルstateを真実源にしない。両端末はリロードで状態を復元できること。
5. **ダブルイリミ進行を手書きしない。** 後述ライブラリに委譲する。

---

## 2. 技術スタック（固定・無断変更禁止）

- React 18 + Vite + TypeScript(strict) + Tailwind CSS
- Supabase（DB / Storage / Realtime）
- 進行ロジック: `brackets-manager`
- 演出: `gsap` / `pixi.js` / `lottie-web`
- 顔検出: `face-api.js` または `@mediapipe/face_detection`
- テスト: `vitest`

> 上記の置き換え・追加ライブラリの導入は、まず理由を提示し確認を取ってから。`package.json` のバージョンはロックし、勝手に上げない。

---

## 3. やってはいけないこと（DON'T）

- ❌ ダブルイリミネーションのWB/LB遷移・bye・GFリセットを自前ロジックで実装する。→ `brackets-manager` を使う。
- ❌ トーナメントのノード座標を定数でハードコードする。→ レイアウト計算関数から取得する。
- ❌ 演出の途中で進行データ(state)を更新する。→ **状態更新と演出再生を分離**する（演出失敗が進行を壊さないため）。
- ❌ 未プリロード素材を再生する。
- ❌ ローカルstateやlocalStorageを状態の真実源にする。→ 真実源はSupabase。localStorageはUI設定など非クリティカルに限定。
- ❌ 勝敗入力を非冪等に実装する（二重送信で状態が壊れる）。→ 適用前に対象matchの状態を検証。
- ❌ 表示端末のためにクロスブラウザ対応コードを書き散らす。→ **表示端末はChrome固定**。前提を活かして簡潔に。
- ❌ 操作端末UIに派手な演出を入れる。→ 操作端末は速さ・誤操作防止が全て。
- ❌ 顔写真を表示サイズに対して過大解像度のまま読み込む。

---

## 4. 状態管理ルール

- 真実源: `events.bracket_snapshot`(JSONB) = `brackets-manager` のexport。
- 操作端末: 勝敗入力 → manager更新 → `export()` → DB保存 → Realtime broadcast。
- 表示端末: マウント時にDBから復元 → Realtimeで更新を購読。
- **undo**: 操作前スナップショットを1手保持。undoは `import()` で巻き戻し。
- 勝敗入力は冪等に。`current_match` と入力対象の整合を必ず検証してから適用。

---

## 5. Realtime 命名規則

- channel: `event:{eventId}`
- broadcast event 名（例）:
  - `match:confirmed` … 勝敗確定（演出トリガ用ペイロード: matchId, winnerTeamId, loserTeamId, 対象ノード座標参照）
  - `bracket:updated` … 状態更新の通知
  - `effect:skip` … 演出スキップ
  - `event:finished` … 優勝確定
- ペイロードは型定義(TS)を必ず用意し、`any` 禁止。

---

## 6. ファイル / フォルダ構成（規約）

```
src/
  routes/          # /admin, /display, /rehearsal
  features/
    registration/  # 撮影・顔トリミング・登録
    draw/          # ランダムペア抽選
    bracket/       # brackets-manager ラッパ + レイアウト計算
    progression/   # 試合進行・勝敗・undo
    presentation/  # 演出(GSAP/Pixi/Lottie/VS画面)
  lib/
    supabase.ts    # クライアント
    realtime.ts    # channel/broadcast ヘルパ
    media.ts       # プリロード
  types/           # 共通型(ブラケット/ペイロード/レイアウト)
  assets/          # 演出素材(命名規則に従う)
```

- `features/bracket` に **レイアウト計算** を集約し、`NodeLayout[]` を返す純粋関数として実装（テスト対象）。
- 演出(`presentation`)は `NodeLayout` を入力として受け取り、座標から描画する。

---

## 7. 命名規則

- 演出素材: `bg_main.webm`, `explosion_alpha.webm`, `vs_template.*`, `placeholder_face.png`。
- Storage顔写真: `participant-faces/{eventId}/{participantId}.jpg`。
- コンポーネント: PascalCase。フック: `useXxx`。型: PascalCase。

---

## 8. 型安全・品質

- TypeScript `strict: true`。`any` は原則禁止（やむを得ない場合はコメントで理由）。
- Supabaseの型は型生成を使い、手書きの不整合を避ける。
- ブラケット/Realtimeペイロード/レイアウトは共通 `types/` に定義。

---

## 9. スタイル / デザイン規約

- 日本語フォントは **Noto Sans CJK JP** を使用（図版・UI共通）。
- 操作端末は実務UI（大ボタン・誤操作防止・undo常設）、表示端末は演出UI、と明確に分離。
- 色だけでWB/LBを区別しない（ラベル併用）。

---

## 10. パフォーマンス規約

- 演出ピーク時に同時に動かす重い要素を絞る（VS表示中は背景動画を停止/減光）。
- 顔写真は表示解像度に最適化してから配置。
- 演出発火前に必要素材のプリロード完了を待つ。

---

## 11. 肖像データ取り扱い

- 顔写真は当日限りの利用前提。イベント終了後に一括削除できる機能/手順を残す。
- 公開URLの露出を最小化（表示端末参照に限定する設計）。

---

## 12. テスト必須箇所

- `features/bracket` の進行ロジック（4/8/16チーム、奇数→bye、GFリセット、undo再入力）。
- レイアウト計算（ノード数・配置に対する座標の妥当性）。
- これらはUIと独立してVitestで担保。**ロジックのバグは本番崩壊に直結するため最優先。**

---

## 13. フェーズ遵守

- `00_REQUIREMENTS.md` のPhase順に実装する。**演出(Phase5)を土台(Phase0-3)より先に作らない。**
- 各フェーズの「Doneの定義」を満たすまで次フェーズの作り込みを始めない。

---

## 14. AIアシスタントへの作業指示

- 変更前に、関連する要件(`00/01/02`)と本ルールに反していないか自己チェックする。
- ライブラリ追加・スタック変更・座標ハードコード・進行ロジック自作を提案しそうになったら、**まず代替（ライブラリ委譲・座標駆動）を検討**し、それでも必要なら理由を添えて確認を求める。
- 不明点は実装を進める前に質問する。推測で進行ロジックや座標を埋めない。

---

## 15. レビュー観点チェックリスト（PR時）

- [ ] 座標がハードコードされていない（レイアウト計算経由）
- [ ] 素材はプリロード後に再生している
- [ ] 真実源はSupabase、UI stateを真実源にしていない
- [ ] 進行ロジックは brackets-manager に委譲している
- [ ] 勝敗入力は冪等、undoが機能する
- [ ] 状態更新と演出再生が分離されている
- [ ] 型に `any` を持ち込んでいない
- [ ] ブラケット/レイアウトのテストがある
- [ ] フェーズ順を守っている
